The right architecture

14 Feb 2024
software development

How many times have you had the opportunity to choose a software architecture? Not many in my case. Most times, the architecture chooses me: you get to a project, the code has some good or bad architecture (probably bad), and you go along with it as well as you can.

Now, I have, of course, had the opportunity to start things from scratch, the proverbial greenfield, the chance to do things The Right Way. I remember one time that I started a project using a reference architecture from Microsoft back in the mid-00s. Microsoft published an archetype to showcase a layered architecture for a .NET application, with lots of boilerplate code, interface, indirection, and all the overengineering needed to make a young developer feel good. But in my experience, these cases are the exception rather than the rule.

“But Javi,” I hear you say, “projects change, and the architecture with them. Today’s monolith is tomorrow’s set of microservices!” Well, most likely, today’s monolith is tomorrow’s distributed monolith, but I see your point.

Why do we care about software architectures?

The first thing to remember when thinking about software architecture is this: there is no such thing as the right architecture for a given project.

Junior software architect’s wet dream

Unlike with buildings, software architectures don’t usually show from the outside. Even for professionals, it can be hard to tell what is behind a given web application. And that is a clue to understanding that the site’s architecture is a means to an end, not an end itself.

And what is the end? Well, imagine somebody giving you a piece of software: a set of docker images, a docker-compose, something like 37signal’s Campfire. You run it, configure it, and start using it. Let’s say that the application works and performs well. How much do you care about its architecture? Not much. You could argue that it is the architecture that makes it work and perform well enough. But ask yourself the following: is it impossible that the application is internally a total mess, with spaghetti code, everything coupled with everything, no unit tests, no comments? No, it is not impossible. And if that were the case, would you care? The answer is only if you’re planning on making changes in the code.

It’s all about the journey

The truth is that software architecture is only important during the building process. Architecture is a misleading word.
